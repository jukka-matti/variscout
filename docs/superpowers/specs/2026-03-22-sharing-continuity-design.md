# Sharing & Continuity UX Design

## Context

VariScout Azure app has solid sharing infrastructure (deep links, Teams SDK integration, OneDrive/SharePoint sync) but the user experience layer is incomplete. Share buttons are scattered, there's no feedback when sharing succeeds, report publishing isn't wired up, and cross-device continuity requires manual steps. This design creates a cohesive sharing story across five areas.

## Scope

Five interconnected improvements for the Azure app (Team plan focus, with Standard plan graceful degradation):

1. Unified Share dropdown in EditorToolbar
2. Report snapshot publishing (HTML + PDF) to SharePoint
3. Self-continuity: "Continue where you left off" Dashboard card + auto-resume
4. Toast feedback for all share actions
5. Better recipient error states for failed deep links

---

## 1. Unified Share Dropdown

### What

Single `Share2` icon button in EditorToolbar with a dropdown menu. Replaces scattered share entry points with one discoverable action.

### Placement

Between Presentation (Maximize2) and Findings toggle (ClipboardList) in the toolbar.

### Menu Items

| Item               | Visibility                    | Action                                                                                              |
| ------------------ | ----------------------------- | --------------------------------------------------------------------------------------------------- |
| **Copy link**      | Always                        | Copies deep link directly to clipboard via `navigator.clipboard` (bypasses Teams SDK even in Teams) |
| **Share in Teams** | In Teams only (`isInTeams()`) | Opens Teams native `sharing.shareWebContent()` dialog with rich preview                             |
| **Publish report** | Report view + Team plan       | Renders HTML snapshot, uploads to SharePoint                                                        |

**Behavior note:** "Copy link" always uses `navigator.clipboard.writeText()` directly — never routes through `useTeamsShare.share()`. This avoids confusion: "Copy link" = clipboard, "Share in Teams" = Teams dialog. The existing `useTeamsShare.share()` fallback chain is used only by "Share in Teams".

### Deep Link Contents ("Copy link")

- Project name (always)
- `chart` param (if focused chart active)
- `finding` param (if finding highlighted)
- `mode=report` (if in report view)
- Filters NOT encoded — they're part of saved project state

### Mobile

Share actions must also appear in the phone overflow menu (`EllipsisVertical`), following the existing pattern where every desktop toolbar action is mirrored in the overflow menu.

### Files to Modify

- `apps/azure/src/components/EditorToolbar.tsx` — add Share dropdown (desktop) + share items in phone overflow menu
- `apps/azure/src/pages/Editor.tsx` — wire handleShareLink, handleShareTeams, handlePublishReport callbacks
- New: `apps/azure/src/components/ShareDropdown.tsx` — dropdown component

### Reuse

- `useTeamsShare` hook (`apps/azure/src/hooks/useTeamsShare.ts`) — for Teams share dialog + setDeepLink
- `buildChartSharePayload`, `buildReportSharePayload` (`apps/azure/src/services/shareContent.ts`)
- `buildFindingLink`, `buildChartLink`, `buildReportLink` (`apps/azure/src/services/deepLinks.ts`)
- `isInTeams()` from `apps/azure/src/teams/`
- Existing dropdown pattern from "Add Data" in EditorToolbar

## 2. Report Snapshot Publishing

### What

One-click "Publish report" renders the current scouting report as a self-contained HTML file and uploads to SharePoint. Anyone in the org can view it in their browser without VariScout.

### Relationship to Existing Markdown Publisher

`usePublishReport` already renders Markdown via `renderReportMarkdown()` and uploads `.md` files to `/VariScout/Projects/` for Copilot Retrieval API indexing (ADR-026). **The HTML snapshot is an addition, not a replacement.** The Markdown publish continues to serve KB search; the HTML snapshot serves human-readable sharing.

### Flow (Two-Phase: Capture then Render)

**Phase 1 — Chart capture (in ReportView.tsx, async):**

1. User clicks "Publish report" from Share dropdown
2. `ReportView.tsx` (which already imports `useChartCopy`) captures each visible chart at fixed export dimensions → produces `Map<chartId, base64PNG>`
3. Captures are awaited sequentially (each needs DOM resize + re-render)

**Phase 2 — HTML rendering (pure function):** 4. Chart base64 strings + report data passed to `renderReportHtml()` in `reportHtmlRenderer.ts` 5. Renders self-contained HTML:

- Report title, process name, date
- KPI grid (Cpk, sigma, sample count)
- Chart images as `<img src="data:image/png;base64,...">` (pre-captured, no DOM needed)
- Hypothesis summary, improvement summary
- Current audience mode (Technical/Summary) preserved
- Inline CSS (no external deps)
- Footer: "Generated by VariScout" + "Open live version" deep link + "Print" button

6. Upload HTML to SharePoint: `/VariScout/Reports/{project-name}-{date}.html`
7. Also trigger existing Markdown publish to `/VariScout/Projects/` (for KB indexing)
8. Toast: "Report published to SharePoint" with "Open" action button

### PDF

- **Existing**: "Save as PDF" via `window.print()` already works in ReportViewBase
- **Published HTML** includes a "Print / Save PDF" button that triggers `window.print()` on the snapshot page
- No PDF library needed

### Constraints

- Team plan only (Standard has no SharePoint). "Publish report" hidden for Standard.
- Same project name + date = SharePoint version history (overwrite, not duplicate)
- `generateReportFilename()` in `reportExport.ts` is currently type-locked to `'md'` extension — extend signature to accept `'md' | 'html'`

### Files to Modify

- New: `apps/azure/src/services/reportHtmlRenderer.ts` — pure function: `(reportData, chartImages: Map<string, string>) => string`
- New: `apps/azure/src/hooks/usePublishHtmlReport.ts` — orchestrates chart capture + HTML render + upload (separate from existing `usePublishReport` to avoid conflating Markdown KB flow)
- `apps/azure/src/services/reportExport.ts` — extend `generateReportFilename` to accept `'md' | 'html'`
- `apps/azure/src/components/views/ReportView.tsx` — wire chart capture + onPublishHtmlReport callback
- `apps/azure/src/pages/Editor.tsx` — add handlePublishReport handler

### Reuse

- `usePublishReport` hook — continues to handle Markdown KB publishing (called alongside HTML publish)
- `useChartCopy` — chart PNG capture at fixed dimensions (called from ReportView.tsx)
- `storage.ts` / `reportUpload.ts` — SharePoint upload functions
- `shareContent.ts` `buildReportSharePayload` for the deep link in footer

## 3. Self-Continuity ("Continue Where You Left Off")

### 3A. Manual: Copy Link (part of Share dropdown)

Already covered in Section 1. "Copy link" captures current view state as a deep link.

### 3B. Automatic: Dashboard Resume Card

#### What

A "Continue where you left off" card appears as the first item in the Dashboard project list, showing the last active project and view.

#### How It Works

- **Write**: When user navigates in the Editor, save a small session marker (debounced ~5s):
  ```typescript
  { project: string, timestamp: string, view?: { chart?: string, findingId?: string, mode?: string } }
  ```
- **Storage**:
  - Team plan: `/VariScout/Projects/.last-session.json` on OneDrive (reuses existing `/VariScout/Projects/` folder — no new root-level path needed)
  - Standard plan: IndexedDB record with key `last-session` (same-device only)
- **Read**: Dashboard checks for the marker on mount. Both reads (OneDrive + IndexedDB dismiss flag) are initiated in parallel. The resume card slot is not rendered until both complete (no layout shift).
- **Display**: If marker exists and is recent (< 24h), show a card at the top of the project list
- **Card content**: Project name, relative timestamp ("2 hours ago"), view context ("I-Chart focused" / "Report view")
- **Click**: Opens Editor with saved view state (same as deep link navigation)
- **Dismiss**: X button sets `lastSessionDismissedFor` in IndexedDB (stores the project name + timestamp). Card is hidden when the dismissed marker matches the current session marker. The flag is implicitly cleared when the session marker updates (i.e., user opens a different project or the timestamp advances beyond the dismissed one).
- **Not shown**: On first-ever launch, or if no projects exist, or while the async read is in progress

#### Cross-Device Flow

1. Save on desktop → `.last-session.json` syncs to OneDrive (via existing `/VariScout/Projects/` folder)
2. Open Teams on phone → Dashboard loads, reads `.last-session.json` from OneDrive
3. "Continue" card shows the desktop project
4. Tap → opens analysis on phone, right where desktop left off

#### Files to Modify

- New: `apps/azure/src/hooks/useLastSession.ts` — read/write last session marker
- New: `apps/azure/src/services/sessionStorage.ts` — OneDrive read/write for `.last-session.json` (small standalone service, not added to the already-large `storage.ts`)
- `apps/azure/src/components/Dashboard.tsx` (or Azure's Dashboard wrapper) — render resume card
- `apps/azure/src/pages/Editor.tsx` — write session marker on navigation changes

## 4. Toast Feedback System

### What

Every share action shows a transient toast notification.

### Architecture

The existing `SyncToastContainer` is **props-driven** — it receives `notifications: SyncNotification[]` from `useStorage()`. It has no imperative `showToast()` API. Rather than force share notifications through the storage layer (wrong conceptual fit), create a new **`useToast` context** at the app level:

- New `ToastProvider` wraps the app (alongside existing providers)
- Exposes `showToast(notification)` via context
- `SyncToastContainer` is refactored to consume from `ToastProvider` (storage notifications are pushed into the same context via `useEffect`)
- Share hooks receive an `onToast` callback option (not a context import — keeps hooks testable)

### Toast Map

| Action                   | Type    | Message                          | Duration   | Action          |
| ------------------------ | ------- | -------------------------------- | ---------- | --------------- |
| Copy link                | success | "Link copied to clipboard"       | 3s         | —               |
| Share to Teams           | success | "Shared in Teams"                | 3s         | —               |
| Teams share dismissed    | info    | "Link copied to clipboard"       | 3s         | —               |
| Publish report           | success | "Report published to SharePoint" | 5s         | "Open" → SP URL |
| Share finding to channel | success | "Finding shared to channel"      | 3s         | —               |
| Any share failure        | error   | "Couldn't share. Try again."     | persistent | "Retry"         |

### Implementation

- `ToastProvider` manages a `notifications` array with `showToast(n)` / `dismissToast(id)` methods
- `SyncToastContainer` unchanged in rendering — still receives `notifications[]` + `onDismiss`, now from `ToastProvider` context instead of props
- Share hooks accept `onToast?: (n: SyncNotification) => void` in their options object (e.g., `UseShareFindingOptions.onToast`)
- `Editor.tsx` passes `showToast` from `useToast()` context into share hook options

### Files to Modify

- New: `apps/azure/src/context/ToastContext.tsx` — `ToastProvider` + `useToast` hook
- `apps/azure/src/components/SyncToast.tsx` — consume from `ToastContext` instead of props (rendering unchanged)
- `apps/azure/src/App.tsx` — wrap with `ToastProvider`
- `apps/azure/src/hooks/useTeamsShare.ts` — add `onToast` option
- `apps/azure/src/hooks/useShareFinding.ts` — add `onToast` option to `UseShareFindingOptions`
- `apps/azure/src/hooks/usePublishHtmlReport.ts` — add `onToast` option
- `apps/azure/src/pages/Editor.tsx` — pass `showToast` to share hook options

## 5. Recipient Error States

### What

When a deep link fails to load a project, show a helpful error message explaining WHY and what to do.

### Error Messages

| Scenario                    | Error code      | Message                                                                                         |
| --------------------------- | --------------- | ----------------------------------------------------------------------------------------------- |
| Project not found           | `not-found`     | "Project not found. It may have been deleted or moved. Ask the person who shared this link."    |
| No channel access           | `forbidden`     | "This project is in a Teams channel you don't have access to. Ask a channel member to add you." |
| Standard plan, team project | `plan-mismatch` | "This project requires a Team plan to access. Contact your admin."                              |
| Offline, not cached         | `offline`       | "You're offline and this project isn't cached locally. Connect to the internet to load it."     |
| Auth expired                | `auth`          | "Your session has expired." + "Sign in" button                                                  |

### Implementation

Extend the existing `SyncErrorCategory` type in `storage.ts` (currently `'auth' | 'network' | 'throttle' | 'server' | 'not_found' | 'unknown'`) with two new categories:

- `'forbidden'` — 403 responses (distinct from 401 `'auth'`). Currently both 401 and 403 are lumped into `'auth'`.
- `'plan-mismatch'` — detected when a deep link references a team-location project but `hasTeamFeatures()` is false.

`classifySyncError()` updated to distinguish 401 (auth expired) from 403 (no access). `loadProject()` checks `hasTeamFeatures()` before attempting team storage, returning `plan-mismatch` if the project can't be found locally.

`Editor.tsx` maps extended error codes to user-friendly messages in the error banner. Error banner includes contextual action buttons ("Sign in", "Go to Dashboard").

### Files to Modify

- `apps/azure/src/services/storage.ts` — extend `SyncErrorCategory` with `'forbidden'` and `'plan-mismatch'`; update `classifySyncError()` for 401 vs 403; update `loadProject()` error returns
- `apps/azure/src/pages/Editor.tsx` — error message mapping + action buttons

## Implementation Order

| Priority | Feature                                | Complexity | Dependencies                             |
| -------- | -------------------------------------- | ---------- | ---------------------------------------- |
| P0       | Toast feedback (Section 4)             | Small      | None — foundation for all other features |
| P1       | Share dropdown (Section 1)             | Medium     | Toast system                             |
| P1       | Recipient error states (Section 5)     | Small      | None                                     |
| P2       | Last session resume card (Section 3B)  | Medium     | None                                     |
| P3       | Report snapshot publishing (Section 2) | Large      | Share dropdown, toast system             |

P0 first because all other features depend on toast feedback. P1 is the core sharing UX. P2-P3 are valuable but independent.
