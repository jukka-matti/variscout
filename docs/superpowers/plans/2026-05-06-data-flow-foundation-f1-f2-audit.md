# F1+F2 Audit

Read-only audit of the VariScout monorepo at branch `main` (current HEAD) to inform the F1 type-level normalization + F2 repository pattern refactor. All file paths below are relative to repo root. Five inventories + a final blockers section.

> Scope decision: this audit lists entities **embedded in or derived from `ProcessHub`** (the F1 `EntityBase` target surface). Project-level wrappers (`ProjectMetadata`, `ProjectRecord`, `SyncItem`, `SyncStateRecord`) are NOT in scope — they are container/sync metadata, not hub-domain entities. They appear in section 6 as a surprise/clarification because the plan does not name them explicitly.

## 5. Action-shape inventory (canvas)

The canonical canvas action union lives at **`packages/core/src/canvas/types.ts`** (NOT in `packages/stores/src/canvasStore.ts` — verify the path: the plan-spec text §"Audit P0.1.5" says "or wherever canvas migration's discriminated-union actions live"; the answer is core, not stores).

`canvasStore.ts` does not directly consume the union — it exposes one method per action shape. The discriminator in `core/canvas/types.ts` is `kind` (NOT `type`). Action builders in `packages/core/src/canvas/actions.ts` produce these shapes for the dispatch pattern, but the store doesn't currently call them. **F2 P2.1's `CanvasAction` consolidation must reuse the existing `kind` discriminator and existing payload field names exactly, OR rename to `type` + remap field names atomically — this is a breaking change to a published-and-consumed union.**

| ---------------------- | ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `PLACE_CHIP_ON_STEP`   | `chipId: string`, `stepId: string`                    | `canonicalMap.assignments[chipId] = stepId`                                                                        | No-op when assignment already matches (returns false in mutator).                                                                                     |
| `UNASSIGN_CHIP`        | `chipId: string`                                      | deletes `canonicalMap.assignments[chipId]`                                                                         | No-op when chip not assigned.                                                                                                                         |

- `hydrateCanvasDocument(snapshot: CanvasHydrationSnapshot)` — bulk replace document state; clears undo/redo. No corresponding action in `core/canvas/types.ts`. F2 must decide whether to model this as an action or keep as a non-dispatched bootstrap.
- `createStepFromChip(chipId, stepName)` — combined ADD_STEP + PLACE_CHIP_ON_STEP. Not in `CanvasAction`. Would correspond to two dispatches OR a new compound action.
- `undo()` / `redo()` / `clearHistory()` — history controls, not document mutations. Live alongside actions in the store but should not be in `CanvasAction`.

## 6. Surprises / blockers

The following surfaced during the audit and should be addressed in the plan **before P1 dispatches**, or at minimum acknowledged in P1.\* task prompts:

1. **`createdAt` shape inconsistency.** Almost every entity uses ISO-8601 strings (`Question`, `SuspectedCause`, `CausalLink`, `EvidenceSource`, `ProcessHub`, `SustainmentRecord`, `ImprovementIdea`, `EvidenceSnapshot.capturedAt/importedAt`). A minority uses `Date.now()` numbers (`Finding`, `ActionItem`, `FindingComment`, `PhotoAttachment`, `CommentAttachment`). Spec D2 dictates `createdAt: number` (Unix ms). **Half the codebase will need a string→number conversion at the seam.** This is a real diff multiplier — recommend P1 either (a) keep ISO strings and amend D2 to `number | string` (loose), (b) convert all to `number` atomically (strict, larger diff), or (c) keep ISO strings on the wire and add a parsed `createdAtMs: number` overlay. Plan author should pick before P1 dispatches.

2. **`CausalLink.hubId` is NOT `ProcessHub['id']`.** It refers to `SuspectedCause['id']` (the "SuspectedCause hub" concept). Shadows the term "hub" used everywhere else. The plan's "typed FK fields" task (P1.4) prescribes `CausalLink.fromFindingId`, `CausalLink.toCauseId` — those don't exist; the actual FK is `CausalLink.hubId` (→SuspectedCause), with `fromFactor` / `toFactor` / `fromLevel` / `toLevel` being **column-name strings**, not entity FKs. This is a fundamentally different shape from what the plan describes. **The plan needs revision before P1.4.**

3. **`EvidenceSourceCursor` is in `apps/azure/src/db/schema.ts`, not `@variscout/core`.** The plan §"Modified — `@variscout/core` types" implies it lives in `processHub.ts` (P1.3). It does not. F1 must decide: (a) move the type into `packages/core/src/evidenceSources.ts`, (b) leave it Azure-only and skip it for `EntityBase`, or (c) leave it Azure-only but apply `EntityBase` there. Option (a) honors ADR-078 D2 best.

4. **`RowProvenanceTag` has no `id` today.** Per ADR-077 D6 it lives in a sidecar `Map<rowKey, tag>` keyed by row identifier. No surrogate id, no createdAt, no FK to its snapshot — `source` and `joinKey` are all it has. F1 P1.3 says "Add `EntityBase`" but that's a substantive shape change (must add `id`, `createdAt`, `deletedAt`, AND a `snapshotId` FK + something to identify the row). Plan should clarify in P1.3.

5. **`OutcomeSpec` has no surrogate id.** It's identified by `columnName` natural key inside `ProcessHub.outcomes[]`. F1 introducing `id` is correct but breaks every fixture and test that constructs hubs by-array. Multi-PR sweep needed; flag in P1.2.

6. **`ProcessMap` and its embedded entities (`ProcessMapNode`, `ProcessMapTributary`, `ProcessMapArrow`, `ProcessMapHunch`).** Spec D3 keeps `canvasState: '&hubId'` as a 1:1 single-row table — implies F1+F2 leaves `ProcessMap` embedded in the hub blob (no surrogate id at the map level; nodes/tributaries/arrows/hunches keep their existing ids inside the embedded record). Plan task P1.5 says "Add `EntityBase` to canvas-related entities" — this is **inconsistent with D3**. Recommend: only `ProcessMapNode` etc. get FK retyping (already have ids); `ProcessMap` itself stays without `EntityBase` until F3 normalization; codify in P1.5 prompt.

7. **`SustainmentRecord.tombstoneAt` IS the existing `deletedAt`.** Phase 6 already shipped soft-delete semantics for sustainment records under a different field name. F1 must either rename `tombstoneAt` → `deletedAt` (atomic refactor across ~12 files in core + azure) or alias / keep both. Plan should explicitly choose.

8. **The plan's framing "stores reach directly into both" is misleading.** As of `main`, `packages/stores/src/{projectStore,investigationStore,improvementStore}.ts` make **zero** Dexie calls. The persistence call sites are in app code (PWA: `App.tsx`, `SaveToBrowserButton.tsx`; Azure: `Dashboard.tsx`, `Editor.tsx`, `useNewHubProvision.ts`, `ProcessHubEvidencePanel.tsx`, sustainment editors). The migration target for "every store WRITE goes through `repo.dispatch`" therefore reduces to "every consumer of Zustand store + persistence helper at the app composition layer dispatches instead." **Re-scope P4 (PWA store migration): the affected files are mostly UI components, not store files.** Same for P6 (Azure). This is good news for diff-size but the plan's phase descriptions need edits.

9. **`wallLayoutStore` imports `dexie` directly** (line 10) and operates a separate Dexie DB (`variscout-wall-layout`). The plan's ESLint guard at P7.2 (`no-restricted-imports` for `dexie` outside `apps/*/src/persistence/`) **will fire on this file**. Mitigations: (a) extend the `ignores` glob to allow `packages/stores/src/wallLayoutStore.ts`, (b) move the WallLayoutDB out of `@variscout/stores` and into both apps' `apps/*/src/persistence/` (dual-implementation), or (c) accept the violation and document it. Plan should pick before P7.2.

10. **A second `Math.random` `generateId` exists at `apps/azure/src/lib/persistence.ts:26`** (not just `findings/factories.ts:24`). D-P3's "fix at the seam" should sweep both. The Azure one is for `SavedProject.id` — pre-production, no migration concern, easy fix.

11. **`addHubComment` in `investigationStore.ts:953` does network-fetch from a store action.** Not Dexie, but still a side-effect leak from the store layer. F2 D1 ("dispatch is the only write path") implies this should also route through repo-dispatch (or stay outside the dispatch layer as a deliberate exception). Plan should clarify: dispatch covers persistence, comment fetch is separate optimistic-update IO.

12. **`PwaDatabase` schema is 2-table only** (`hubs`, `meta`). Azure's is 12+ tables across 8 versions. F2 PR2's `PwaHubRepository.dispatch` therefore loads/rewrites a single blob (per D-P2 "delegate to existing storage"); F2 PR3's `AzureHubRepository.dispatch` will fan out across the existing per-table writes. This is fine, but the PR sizes will be very different — PR2 is small, PR3 is large. Plan should reflect.

13. **Sustainment + ControlHandoff are hub-domain entities** (lifecycle bound to hubs/investigations) and the spec lists them under "stores in scope", but the plan's P1 entity list doesn't name them in the enumeration of types getting `EntityBase`. Section 1 above adds them. Plan P1 sub-tasks should explicitly cover them.

14. **`InvestigationCategory` factory does not set `createdAt`.** Spotted while writing section 2. Trivial but the plan-cited "all factories already set `createdAt`" assumption is not strictly true.

15. **Project-level types (`ProjectRecord`, `SyncItem`, `SyncStateRecord`, `SavedProject`, `ProjectMetadata`) are NOT in scope for F1+F2** per the spec's hub-centric framing. `apps/azure/src/lib/persistence.ts` (READ + WRITE on `db.projects`) and `apps/azure/src/services/cloudSync.ts` (sync overlay) keep using Dexie directly. The ESLint guard at P7.2 must allow these unless F1+F2 deliberately covers project-level too. Recommend explicit "project-level out of scope" callout in the plan.
