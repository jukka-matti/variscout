# ADR-042: Project Dashboard

**Status:** Accepted
**Date:** 2026-03-22

## Context

When users reopen a saved VariScout project, they land directly in the Editor with the last saved state restored silently. There is no orientation moment — no summary of where things stand, what hypotheses have been tested, or what actions are overdue. Users reopen projects for specific reasons (continue analysis, check a hypothesis, add new data, log an improvement idea), but the tool does not acknowledge or support those intents.

Additionally, CoScout cannot search the project's investigation history. If a user asks "have we already checked if it's the material?", CoScout has no tool to search findings or hypotheses — it can only work with whatever is in the current AI context window. The gap is discoverability, not data model structure; the hypotheses and findings already exist.

## Decision

### 1. Project Dashboard — persistent peer view

Add a **Project Dashboard** as a persistent peer view alongside the analysis Editor within a loaded project. It is toggled via `activeView: 'dashboard' | 'editor'` in `panelsStore`.

- **Default landing** for saved projects with data (unless a deep link is present)
- **Always accessible** via an "Overview" tab; "Edit" tab switches back to the Editor
- **New projects** (no data) skip straight to the Editor in FRAME mode
- **Deep links** (Teams task links, `initialFindingId`, `initialChart`) bypass the dashboard and open the Editor directly at the target
- **Progressive enhancement** — the dashboard works without AI; the AI Summary Card is hidden when AI is unavailable and the layout adapts to full-width status

The dashboard layout has three zones:

- **Project Status (left)** — project name, last edited timestamp, journey phase indicator (FRAME/SCOUT/INVESTIGATE/IMPROVE), current focus derived from `ViewState`, findings by status (clickable counts), hypothesis tree summary with validation status and η² values, and action progress bar. All items are clickable and switch to the Editor with the appropriate panel or view pre-opened.
- **AI Summary Card (right top)** — 1–3 sentence contextual summary generated by `buildDashboardSummaryPrompt()`, refreshed on each return to the dashboard when project state has changed. Includes a "Ask CoScout..." quick-ask input that stores the question in `aiStore.pendingDashboardQuestion`, switches to the Editor, and opens the CoScout panel with the question pre-loaded.
- **Quick Actions (right bottom)** — contextual buttons (Go to analysis, Add new data batch, View report, Review actions) based on project state.

The dashboard reads from the same `DataContext` as the Editor. No data is duplicated. The AI summary is the only async call.

### 2. `search_project` CoScout tool — read, auto-execute

A new CoScout tool that searches findings, hypotheses, improvement ideas, and actions in the current project by text and optional filters.

- Pure function `searchProjectArtifacts()` in `packages/core/src/ai/searchProject.ts`
- Case-insensitive substring match across all text fields
- `finding_status` and `hypothesis_status` filters are independent — status filters only apply to their respective artifact types; other types are included unfiltered when `artifact_type` is `'all'`
- Results sorted by exact match > starts-with > contains, then by recency
- **Max 5 results** to stay within token budget (~250–400 tokens)
- No embeddings — project data is small (max ~30 hypotheses, ~50 findings per project)
- **Phase-gating**: available in SCOUT, INVESTIGATE, IMPROVE; not in FRAME (no artifacts to search)

### 3. `navigate_to` CoScout tool — hybrid execution

A new CoScout tool that navigates to a specific finding, hypothesis, chart view, workspace, or the dashboard itself.

- **Auto-executes** when `restore_filters` is `false` (default) — panel/view switching is low-risk and reversible
- **Shows proposal card** when `restore_filters` is `true` — filter restoration mutates the data pipeline state and requires user confirmation. Reuses the `ActionProposalCard` pattern from ADR-029 and `computeFilterPreview()` from `actionTools.ts`
- `'dashboard'` is a valid target — CoScout can navigate back to the Project Dashboard on request
- **Phase-gating**: available in SCOUT, INVESTIGATE, IMPROVE; not in FRAME

### 4. Tool schema constraints

Both new tools use strict mode: `additionalProperties: false` on all schemas, no `default` values in JSON schemas (defaults handled in handler code), `required` arrays explicitly declared.

### 5. Store extensions

Four existing Zustand stores receive new fields to support dashboard navigation:

| Store                | New field                                  | Purpose                                              |
| -------------------- | ------------------------------------------ | ---------------------------------------------------- |
| `panelsStore`        | `activeView: 'dashboard' \| 'editor'`      | Track current top-level view; persisted in ViewState |
| `findingsStore`      | `statusFilter: FindingStatus \| null`      | Filter findings panel by status from dashboard click |
| `investigationStore` | `expandedHypothesisId: string \| null`     | Scroll-to and highlight target from dashboard click  |
| `aiStore`            | `pendingDashboardQuestion: string \| null` | Pre-load question from dashboard "Ask CoScout..."    |

`activeView` is included in `ViewState` for persistence (restores on project reopen). Default is `'editor'` for backward compatibility.

### 6. Core utilities

Two new functions in `@variscout/core`:

- `searchProjectArtifacts()` in `packages/core/src/ai/searchProject.ts` — pure search function
- `buildDashboardSummaryPrompt()` in `packages/core/src/ai/prompts/dashboardSummary.ts` — prompt builder for the AI summary card. Uses fast tier (same as narration). Cache key is a hash of `findingCount + hypothesisStatusCounts + actionCompletionCount` — invalidates when project state changes meaningfully.

### 7. New components (Azure-only)

Three new components in `apps/azure/src/components/`:

| Component              | Purpose                                        |
| ---------------------- | ---------------------------------------------- |
| `ProjectDashboard`     | Full dashboard view (top-level peer to Editor) |
| `ProjectStatusCard`    | Status summary (findings, hypotheses, actions) |
| `DashboardSummaryCard` | AI summary card + quick-ask input              |

## Consequences

### Positive

- **Orientation on reopen** — users see where they left off without navigating through panels manually. Serves OpEx Olivia (KPI monitoring), Green Belt Gary (investigation tracking), and Field Fiona (mobile status check) equally.
- **CoScout can answer history questions** — "have we checked X?" triggers `search_project` automatically, grounding the answer in actual project data.
- **Navigation by intent** — `navigate_to` lets CoScout direct users to relevant views after search, closing the loop from question to context.
- **Progressive enhancement** — the status view works fully offline and without AI; the AI card degrades gracefully.
- **No new data model** — all status data comes from already-loaded `AnalysisState`. Hypotheses and findings are the right containers; no new entity types are needed.

### Negative

- **Tool count: 13 → 15** — still well within OpenAI function calling limits; phase-gating keeps the active tool set small.
- **`activeView` in ViewState** — adds a new field to the persistence schema. Backward compatibility: `undefined` maps to `'editor'` on load.
- **`expandedHypothesisId` in `investigationStore`** — promotes tree expand/collapse logic from `HypothesisTreeView` component state to store state. A small increase in store scope but necessary for cross-feature navigation.
- **Dashboard is Azure-only** — PWA has no project persistence and no dashboard. This is the correct boundary.
- **AI summary adds one async call** — mitigated by the state-aware cache key (no refetch if state is unchanged) and the fast model tier.
