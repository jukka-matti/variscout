# ADR-059: Web-First Deployment Architecture

**Status**: Accepted

**Date**: 2026-04-02

**Supersedes**: ADR-016 (Microsoft Teams Integration), ADR-018 (Channel @Mention Workflow)

**Related**: ADR-007 (Marketplace Distribution), ADR-033 (Pricing Simplification), ADR-058 (Deployment Lifecycle)

---

## Context

VariScout is deployed into customer Azure subscriptions via Azure Marketplace as a Managed Application. The current architecture (ADR-016) positions VariScout as a Microsoft Teams app for the Team tier, requiring 5 admin-consent Graph API permissions:

- `Files.ReadWrite.All` — OneDrive/SharePoint file sync
- `Channel.ReadBasic.All` — channel drive resolution
- `ChannelMessage.Send` — Adaptive Cards in channels
- `Sites.Read.All` — Knowledge Base search
- `People.Read` — people picker for task assignment

### Problem

Enterprise manufacturing customers — VariScout's target market — have strict IT governance:

1. **Permission perception**: `Files.ReadWrite.All` sounds like "read all SharePoint files" to a CTO, even though delegated scope limits to user's own access
2. **Admin consent friction**: Tenant admin sign-off required before any Team tier user can start — often a weeks-long approval process
3. **Third-party app resistance**: Installing a third-party app in the M365 tenant triggers security reviews, especially as Microsoft tightens governance (Copilot content surfacing, EEEU retirement, stricter app vetting 2025-2026)
4. **Unnecessary coupling**: Every Teams-specific feature has a simpler web equivalent that requires no special permissions

### Key Finding

None of the Teams integration features are irreplaceable:

| Teams Feature        | Web Equivalent                                 | Permission Impact              |
| -------------------- | ---------------------------------------------- | ------------------------------ |
| Channel file storage | Azure Blob Storage (customer's resource group) | Azure RBAC (no Graph API)      |
| Teams camera capture | Browser `<input capture="environment">`        | None                           |
| Adaptive Cards       | Teams Incoming Webhook (future)                | None (customer-configured)     |
| Teams SSO + OBO      | EasyAuth only                                  | `User.Read` (no admin consent) |
| App icon in Teams    | Static tab (URL bookmark)                      | None                           |

---

## Decision

**Shift VariScout from a "Teams app" to a "web application deployed via Azure Marketplace."**

### Architecture

- **Auth**: EasyAuth only (`User.Read` — zero admin consent for either tier)
- **Team storage**: Azure Blob Storage in customer's resource group (replaces OneDrive/SharePoint)
- **Photo capture**: Browser `<input type="file" accept="image/*" capture="environment">`
- **Teams presence**: Optional static tab (documentation, not code dependency)
- **Teams SDK**: Removed entirely from codebase

### Tier Model (Preserved)

- **Standard (€79/month)**: Solo analyst. Local storage (IndexedDB + `.vrs` files). `User.Read` only.
- **Team (€199/month)**: Collaboration. Shared Blob Storage + team features. `User.Read` + `People.Read` (both user-consent, no admin consent).

### Blob Storage Architecture

Team tier projects are stored in an Azure Blob Storage container provisioned in the customer's resource group:

```
variscout-projects/
├── {projectId}/
│   ├── analysis.json
│   ├── metadata.json
│   └── photos/{findingId}/{photoId}.jpg
└── _index.json
```

Browser accesses Blob Storage via SAS tokens generated by a server-side endpoint (`/api/storage-token` on the App Service). Tokens are container-scoped, time-limited (1hr), and require valid Entra ID authentication.

Conflict resolution: ETag-based optimistic concurrency on `metadata.json`.

### Permission Comparison

| Scope                   | Before (Team)     | After (Standard) | After (Team)       |
| ----------------------- | ----------------- | ---------------- | ------------------ |
| `User.Read`             | Delegated         | Delegated        | Delegated          |
| `People.Read`           | Delegated         | —                | Delegated          |
| `Files.ReadWrite.All`   | **Admin consent** | —                | —                  |
| `Channel.ReadBasic.All` | **Admin consent** | —                | —                  |
| `ChannelMessage.Send`   | **Admin consent** | —                | —                  |
| `Sites.Read.All`        | **Admin consent** | —                | —                  |
| Azure Blob RBAC         | —                 | —                | Yes (Azure-native) |

**Result: Zero admin-consent Graph API permissions.**

## Consequences

### What becomes easier

- **Enterprise adoption**: No admin consent, no security review for Graph permissions, no third-party app installation in M365 tenant
- **Sales pitch**: "We only ask for your name and email" — strongest possible trust story
- **Codebase simplicity**: Remove Teams SDK (~40KB), OBO Azure Function, dual auth paths, Teams manifest, Teams context detection
- **Testing**: One auth flow to test (EasyAuth) instead of three (EasyAuth + Teams SSO + OBO fallback)
- **Non-Microsoft future**: Blob Storage pattern can later support AWS S3 or GCP Cloud Storage

### What becomes harder

- **Teams presence**: App no longer appears natively in Teams — requires manual static tab setup by customer IT
- **Channel notifications**: No automatic Adaptive Cards; webhook notifications require customer configuration (deferred to future phase)
- **Knowledge Base**: SharePoint search (ADR-026) is deferred — needs alternative approach without `Sites.Read.All`
- **Mobile camera**: Browser `<input capture>` is reliable but less polished than Teams SDK camera (no in-app preview, relies on OS camera)

### What is deferred

- **Knowledge Base reimplementation**: Needs new backend approach (Blob Storage search, Azure AI Search, or customer-managed)
- **Webhook notifications**: Teams Incoming Webhook for channel alerts — Phase 2 feature
- **Email notifications**: For non-Teams customers — Phase 3 feature

## Implementation

### Infrastructure Changes

| Component                       | Action                                                                   |
| ------------------------------- | ------------------------------------------------------------------------ |
| App Registration                | Remove admin-consent scopes; keep `User.Read` + `People.Read`            |
| ARM template (`main.bicep`)     | Add Storage Account + container + RBAC; remove Function App              |
| Azure Function (token-exchange) | Remove entirely                                                          |
| App Service `server.js`         | Replace `/api/token-exchange` with `/api/storage-token` (SAS generation) |

### Code Changes

| Component                              | Action                                      |
| -------------------------------------- | ------------------------------------------- |
| `apps/azure/src/teams/`                | Remove entirely                             |
| `apps/azure/src/auth/graphToken.ts`    | Remove OBO flow; simplify to EasyAuth only  |
| `apps/azure/src/services/cloudSync.ts` | Replace Graph API with Blob Storage SDK     |
| `@microsoft/teams-js` dependency       | Remove from `package.json`                  |
| Photo capture components               | Replace Teams camera with `<input capture>` |
