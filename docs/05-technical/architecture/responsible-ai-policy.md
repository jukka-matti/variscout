# Responsible AI Policy — VariScout

Single reference for VariScout's Responsible AI posture. Covers all AI surface areas, guardrails, data classification, and development process governance.

## §1 — Principles

VariScout's AI integration follows the principles documented in [`aix-design-system.md`](aix-design-system.md):

1. **Deterministic primacy** — AI explains computed results, never generates numbers
2. **Stats-only payloads** — Raw measurements never enter AI context
3. **Proposal-not-execution** — Action tools return proposals requiring user confirmation
4. **Tenant isolation** — AI runs in the customer's Azure subscription
5. **Graceful degradation** — All AI features fail silently; the tool works without AI

## §2 — AI Surface Area Map

### Data that enters AI

| Data type            | Source                         | Example                                |
| -------------------- | ------------------------------ | -------------------------------------- |
| Computed statistics  | `buildAIContext()`             | Mean, StdDev, Cpk, pass rate, ANOVA η² |
| Analyst-written text | Finding descriptions           | "Machine A shows Cpk 0.85"             |
| Analyst-written text | Hypothesis text                | "Nozzle wear causes drift"             |
| Analyst-written text | Action items                   | "Replace nozzle tip weekly"            |
| Analyst-written text | Process descriptions           | "Filling line 3, 500ml bottles"        |
| Analyst-written text | Problem statements             | "Cpk below 1.33 target"                |
| Analyst-written text | Outcome notes                  | "Cpk improved to 1.45"                 |
| CoScout conversation | Last 10 messages               | User questions + AI responses          |
| KB document snippets | Foundry IQ / Remote SharePoint | 400 chars max per document             |
| Factor names         | CSV column headers             | "Machine", "Operator", "Shift"         |
| Category values      | CSV data values                | "Machine A", "Morning", "Line 1"       |

**Never sent to AI:**

- Raw measurement values (only summary statistics)
- Photos (stored in OneDrive, never sent to AI)
- User credentials or tokens
- Other users' data (tenant-isolated)

## §4 — PII & Data Classification

### Design decision

Analyst-written text is sent to AI without client-side PII scrubbing.

**Rationale:**

- The analyst is an authenticated data owner in the customer's Azure AD tenant
- Text is deliberately authored by the analyst (not passively collected)
- AI runs in the customer's own Azure subscription (tenant-isolated)
- Azure OpenAI content filters apply server-side (DefaultV2 policy)
- No data leaves the customer's Azure region

### Knowledge Base documents

- Customer-controlled SharePoint folder; per-user permissions via delegated token passthrough
- Customers choose which folders are searchable (AdminKnowledgeSetup)
- Document snippets are truncated to 400 characters maximum
- Source attribution required in AI responses (`[Source: name]` notation)

### Telemetry

Only non-PII metadata is recorded:

- Feature name (narration, insight, coscout)
- Model name and deployment
- Latency (ms) and token counts (prompt, completion)
- Error type (if applicable)

No prompt content, user text, or AI response content is sent to Application Insights.

### Upgrade path

If enterprise customers require client-side PII detection:

- Azure AI Content Safety PII Detection API can be added as a pre-processing step
- This would scan analyst-written text before it enters AI context
- Currently not implemented — design decision documented here for future reference

## §5 — Knowledge Base Guardrails

Foundry IQ retrieval (Remote SharePoint) has these specific controls:

| Control              | Implementation                                                              |
| -------------------- | --------------------------------------------------------------------------- |
| Per-user permissions | Delegated token passthrough — user can only search docs they have access to |
| Folder scoping       | KQL filter on SharePoint path — only configured KB folders are searched     |
| Snippet truncation   | 400 chars max per document — limits context window exposure                 |
| Source attribution   | `[Source: name]` citation required in CoScout responses                     |
| Customer control     | Admin selects which folders are KB sources (AdminKnowledgeSetup)            |

### Recursive knowledge risk

Published reports could theoretically be discovered and retrieved in future analyses. Mitigations:

- Reports are stored in `/VariScout/Projects/` (separate from KB source folder)
- Customers control which folders are KB sources
- Retrieval uses per-user permissions — only authorized users see results

## §6 — Development Process

### Prompt change governance

1. Read [AIX Design System](aix-design-system.md) principles
2. Edit prompt template in `packages/core/src/ai/prompts/`
3. Run prompt safety tests: `pnpm --filter @variscout/core test`
4. Verify with coffee dataset (n=24 triggers hedging)
5. Document rationale in commit message

### New AI feature checklist

- [ ] What new data enters AI context? Document in §2 above.
- [ ] Does the feature share AI output externally? If yes, require user confirmation (proposal pattern).
- [ ] Does the feature add new tools? Ensure `strict: true` + `additionalProperties: false` + phase-gating.
- [ ] Run prompt safety tests after changes.
- [ ] Update `ai-safety-report.md` if the surface area changes.

### Automated safety tests

The prompt safety test suite (`packages/core/src/ai/__tests__/promptSafety.test.ts`) verifies:

1. **Safety instruction presence** — All system prompts contain "Never invent data" and `TERMINOLOGY_INSTRUCTION`
2. **Context injection resistance** — Adversarial finding/hypothesis/process text is embedded as literal data, not at prompt level
3. **Tool schema strictness** — All 13 CoScout tools have `strict: true` and `additionalProperties: false`

These tests run as part of `pnpm --filter @variscout/core test`.
