# `wallLayoutStore` is `STORE_LAYER = 'annotation-per-project'`, not `'view'`. `selection: Set<NodeId>` is a per-session view-state field within that annotation store; it is intentionally OMITTED from `WallLayoutSnapshot` to dodge the Set/JSON Dexie round-trip hazard

Correcting an earlier `docs/investigations.md` claim that the store itself was view-layer. The annotation classification is correct (zoom / pan / nodePositions are durable per-project annotations); selection is the lone exception inside the store and stays transient by exclusion at the persist boundary. Locked with the regression test in `packages/stores/src/__tests__/wallLayoutStore.test.ts` (selection persistence boundary block) + the docstring above `interface WallLayoutSnapshot` in `packages/stores/src/wallLayoutStore.ts`. If a future spec needs persistent multi-select recall, convert at the boundary (`partialize: { ...s, selection: [...s.selection] }` + rehydrate `new Set(raw.selection)`) ā€” do NOT add `selection: Set<NodeId>` to `WallLayoutSnapshot` directly. _Pinned 2026-05-08._ Source: PR #143 + `docs/investigations.md` "wallLayoutStore.selection Set/JSON Dexie round-trip" resolution.

- **ADR-073 ā€” No statistical roll-up across heterogeneous units.** Capability indices computed against different `SpecRule`s cannot be combined arithmetically (Watson's locality rule). Enforced by **structural absence**, not by permission predicate: the engine does not expose `meanCapabilityAcrossHubs` or any named variant; CI guard tests in `packages/core/src/__tests__/architecture.noCrossInvestigationAggregation.test.ts` enforce the absence. Distributions, not aggregates. _Closed 2026-04-29._ Source: [`docs/07-decisions/adr-073-no-statistical-rollup-across-heterogeneous-units.md`](07-decisions/adr-073-no-statistical-rollup-across-heterogeneous-units.md). Generalizes to any heterogeneity dimension (different specs, contexts, products, suppliers, shifts) ā€” not just process families. See `feedback_aggregation_heterogeneous_specs.md`.

- **ADR-075 ā€” PWA atomic deploy + update policy.** Four-layer response to a user-reported live-deploy regression where a stale SW-cached `index.html` referenced chunk hashes that no longer existed. (1) SW update mode is `autoUpdate` with `skipWaiting: true`, `clientsClaim: true`, `cleanupOutdatedCaches: true` ā€” auto-claims existing users on next visit (see ADR-075 Amendment 2026-05-02 below for why the original prompt-mode shape was reverted). (2) Every `React.lazy()` site wraps in `lazyWithRetry`, which on chunk-load error evicts `workbox-precache-*` and reloads once-per-session (`sessionStorage` guarded against Safari private mode); `PasteScreen` idle-prefetched so the failure self-heals before user click. (3) Vercel headers: `vercel.live` + Pusher allowed in CSP (`script-src` + `connect-src` + `frame-src`), explicit MIME for `/assets/*`, immutable cache for `/assets/*`, `no-cache` for `/index.html` + `/manifest.webmanifest`. (4) `scripts/check-dist-integrity.mjs` blocks any PWA build whose `dist/index.html` references missing chunks, run inside `scripts/pr-ready-check.sh`. _Closed 2026-05-02._ Source: [`docs/07-decisions/adr-075-pwa-atomic-deploy-and-update-policy.md`](07-decisions/adr-075-pwa-atomic-deploy-and-update-policy.md). Cross-references ADR-058 (Azure-app deployment lifecycle) and ADR-059 (web-first).

- **PWA SW prompt-mode reverted (2026-05-02)** ā€” Reverted to `autoUpdate + skipWaiting:true` two days after ADR-075's prompt-mode introduction. Reason: existing users with the old SW have no path to the new SW's update banner because the banner UI lives in NEW code, but old SW serves cached OLD HTML. `lazyWithRetry` (still active) is the actual defensive layer. ADR-075 amended in place; `UpdatePrompt`, `swUpdates`, and `workbox-window` deleted; `frame-src 'self' https://vercel.live` added to CSP for the Vercel Live preview-comments iframe. _Closed 2026-05-02._ Source: [`docs/07-decisions/adr-075-pwa-atomic-deploy-and-update-policy.md`](07-decisions/adr-075-pwa-atomic-deploy-and-update-policy.md) Amendment.

- **ADR-074 ā€” SCOUT level-spanning surface boundary policy.** Each surface owns exactly one level of the three-level methodology (L1 outcome / L2 flow / L3 mechanism) and lenses the other two by linking to the surface that owns each level ā€” never by re-rendering or recomputing. SCOUT owns L1 outcome reading; FRAME owns L2 authoring; the Hub Capability tab owns L2 reading; Investigation Wall owns L3 hypothesis canvas; Evidence Map owns L3 factor network; INVESTIGATE owns L3 case-building. Same enforcement mechanism as ADR-073: structural absence + CI guards, not permission predicates. Prevents the four concrete temptations during implementation (SCOUT redoing column mapping, INVESTIGATE recomputing outcome stats, Hub Capability tab implementing its own SuspectedCause UI, Evidence Map maintaining its own boxplots). _Closed 2026-04-29._ Source: [`docs/07-decisions/adr-074-scout-level-spanning-surface-boundary-policy.md`](07-decisions/adr-074-scout-level-spanning-surface-boundary-policy.md); companion design at [`docs/superpowers/specs/2026-04-29-multi-level-scout-design.md`](superpowers/specs/2026-04-29-multi-level-scout-design.md).
