# Canvas Wall overlay (PR8 sub-PR 8e) brainstorm locked: Fork 1 + V1 read-only + viewport-adaptive mobile

Architectural fork chosen pre-brainstorm per `feedback_honor_vision_commitments`: embed `WallCanvas` viewport as canvas overlay layer (vision §5.6 verbatim), not the hybrid expand-to-modal or status-quo+spec-amend hedges. Six edge-case decisions locked at [`docs/superpowers/specs/2026-05-08-canvas-wall-overlay-design.md`](superpowers/specs/2026-05-08-canvas-wall-overlay-design.md): V1 is **read-only Wall mirror with click-to-drill** (Q1 — overlay skips DnD context, ⌘K palette, hub-comment SSE, `MissingEvidenceDigest` HTML panel; click handlers route to navigate-and-select in destination view; mutating affordances stay in destination per §5.6 cadence-scan vs deep-work split); toggle **hidden until ≥1 hub/question/finding** (Q2, per `feedback_hidden_vs_disabled_cta` — Wall has its own EmptyState component which would be jarring as an overlay over a populated canvas; the existing four overlays are unaffected since they project no-op badges, not empty surfaces); **shared `useWallLayoutStore` viewport** (Q3 — pan/zoom/groupByTributary mutations in either surface mirror live, extends "same data, two views" to view-state, per-project annotation layer already exists); **input layering canvas → Wall (`pointer-events: none` while draw-hypothesis active) → tool gesture handler** (Q4, extends 8d Spec 4 §6 input precedent — the cadence-scan + draw-hypothesis combined flow is real and valuable, mutual-exclusion and Wall-always-on-top alternatives both kill it); **`'wall'` ∈ `activeCanvasOverlays` is View layer** (Q5 — symmetric with the four existing overlays; promoting to per-user persistence would make Wall the lone outlier or scope-creep into all five overlays); **viewport-adaptive mobile fallback** (Q6 — ≥768px renders the overlay layer, <768px re-skins the toolbar slot to "Open Wall ↗" navigation button that deep-links to Investigation tab → Wall sub-tab; grounded in industry pattern across Miro/FigJam/Mural/Figma mobile/Power Apps responsive design — layered overlays are a desktop-canvas affordance, mobile collapses to sequential views). Implementation: new `'wall'` value on `CanvasOverlayId` union, new `<CanvasWallOverlay/>` internal component, new `mode?: 'destination' \| 'overlay'` prop on `WallCanvas` (purely subtractive — no destination-view callsite migration), new `useHasInvestigationContent()` predicate, new `WallShortcutButton` for mobile, new `onOpenWall` prop on `<Canvas/>`. **No new persistent state**, no `STORE_LAYER` changes, `layerBoundary.test.ts` adds the `'wall'` overlay assertion. **Out of scope for 8e:** write affordances inside the overlay (DnD, ⌘K, comments, card reorder), 8f canvas viewport architecture, retroactive Spec 4 consolidation doc, animation polish for overlay mount/unmount. Implementation plan to be written in a fresh session per `superpowers:writing-plans`; ~6 tasks per master plan §4 8e; subagent-driven execution per `feedback_subagent_driven_default` with one worktree per parallel agent (`feedback_one_worktree_per_agent`). Closes the canvas PR4c-PR6 retrospective entry _"Canvas Wall overlay is badge projection, not 'same data, two views' mirror (vision §5.6)."_ _Pinned 2026-05-08._
