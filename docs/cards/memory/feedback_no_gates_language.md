Do not use "gate" or "gates" when describing methodological steps, validation points, or workflow checkpoints in VariScout.

**Why:** When proposing methodology-correctness work (MSA, power, question-data fit), I described these as "methodological gates" the analyst would pass through. The user pushed back: "i am not a fan of gates! that got me a bit scared". The framing reads as restrictive / blocking / mandatory — the opposite of how VariScout positions methodology, which is integrated, supportive, and analyst-driven (per `feedback_pwa_philosophy.md`'s "guided frustration is pedagogy" — guidance, not gating).

**How to apply:** When discussing validation steps, MSA, sample size, power, capability CIs, question-data fit, or any methodology-correctness work, use:
- "guidance", "support", "scaffolding"
- "checkpoint" (the analyst chooses to look)
- "step" (sequential but not blocking)
- "lens" (additive view)
- "affordance" (UI element that enables)

Never: "gate", "gates", "gating", "gated".

This applies in conversation, in spec writing, in ADRs, and in user-facing copy.
